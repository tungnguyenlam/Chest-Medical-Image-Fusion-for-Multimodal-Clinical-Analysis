## 2026-05-10 ‚Äî make `camchex/main.py` runnable from project root

**Goal.** User reported `python camchex/main.py fit --config camchex/config.yaml --config camchex/config.local.yaml` from the project root crashes in `pd.read_csv` because `data.datamodule_cfg.train_df_path: ../data/data-camchex/03_mimic_train.csv` (and image paths like `images/p1x/...` in CSVs) are all written relative to `camchex/`, not the project root. They want a single command that works from project root without rewriting configs.

**Changes.**
- `camchex/main.py:1-22` ‚Äî at module import time, walk `sys.argv` and rewrite any relative `--config <path>` / `--config=<path>` values to absolute paths against the original cwd, then `os.chdir()` into the script's own directory before constructing `LightningCLI`. Imports of `torch`/Lightning/`model.wrapper`/`dataset.datamodule` moved below the chdir block (cosmetic ‚Äî they don't depend on cwd, but keeps the path-fixing logic at the top where it's visible).

**Reasoning.** Three candidates considered:

1. **Rewrite every path in `config.yaml` to be project-root-relative** and update `dataset/dataset.py` to resolve image paths against the project root. Touches the most files, breaks the existing `cd camchex && bash train.sh` workflow documented in `camchex/CLAUDE.md`, and `train.sh` itself launches with cwd=`camchex/` (`#SBATCH --output=../output/...`, `mkdir -p ../output/...`). Would force a coordinated change across SLURM scripts, rsync workflows, and per-machine `config.local.yaml` files on four machines.
2. **Wrapper script at project root** that just `cd`s into camchex and re-execs python. Works but adds a new entry point, doesn't help users who type the python command directly (which is what the user did).
3. **Chdir inside `main.py`** (chosen). One-file change, preserves the load-bearing assumption that cwd=`camchex/` at training time (config relative paths, CSV image paths, `mkdir -p ../output/...` in `train.sh`, callbacks importable as `callbacks.ema_callback.EMACallback` because `sys.path[0]` is the script dir). The `train.sh` flow is unaffected because it already cd's into camchex/ before invoking python, so the chdir is a no-op there.

The argv-rewriting step is necessary because `--config camchex/config.yaml` (the form the user typed) is project-root-relative; if we just chdir without rewriting, jsonargparse opens it relative to camchex/ and fails. Rewriting only `--config` (not arbitrary `--data.datamodule_cfg.train_df_path` overrides) is intentional ‚Äî those latter values live inside the YAML and are already camchex-relative; rewriting them would break the YAML semantics.

**Assumptions.**
- `sys.path[0]` is set by Python to the directory containing `main.py` regardless of cwd, so the `from model.wrapper import ...` imports keep working after chdir. (True for `python camchex/main.py` invocation; would NOT be true for `python -m main` from project root, but that's not how the project is launched.)
- `config.local.yaml` doesn't exist on `kubuntu` (the dev machine where this edit was authored), so I couldn't smoke-test the exact failing command end-to-end. The error in the user's traceback was specifically `FileNotFoundError` on the train CSV path during `CaMCheXDataModule.__init__`, which is downstream of config parsing ‚Äî confirming the config files themselves were found and the only remaining issue is the relative paths inside them. Chdir resolves that.
- All callsites that read paths from config do so via `pd.read_csv(...)` / `Image.open(...)` with no path normalization ‚Äî verified by inspecting `dataset/datamodule.py:16` in the traceback. So plain chdir is sufficient; no path rewriting in the data layer needed.

**Gotchas.**
- The chdir runs at **import time**, before `if __name__ == "__main__"`. This is deliberate ‚Äî `LightningCLI` parses argv inside its constructor, which is called from `cli_main()`, so the chdir must happen before that. But it means importing `camchex.main` as a module from elsewhere (e.g. tests) would have the side effect of changing cwd. Acceptable since nothing imports `main.py`; flag if that changes.
- Argv rewriting only handles `--config`. If a future user passes other path-bearing flags from project root (e.g. `--trainer.default_root_dir ../output/...`), those won't be rewritten and will resolve against camchex/ post-chdir. Not fixing speculatively.
- `train.sh` is unchanged. It still does `cd "$SLURM_SUBMIT_DIR"` and assumes the user submitted from `camchex/`. The new chdir in `main.py` is idempotent in that flow.
- `os.path.abspath(os.path.join(_ORIG_CWD, p))` ‚Äî `os.path.join` discards `_ORIG_CWD` if `p` is already absolute, but we guard with `os.path.isabs(p)` first anyway for clarity.

**Follow-ups.**
- Smoke-test on the user's server (`hngoc@DESKTOP-UUC7V3K`) ‚Äî they have `config.local.yaml` and the data files; this kubuntu machine doesn't.
- Consider documenting the new project-root invocation in `camchex/CLAUDE.md` ("Training" section) alongside the existing `cd camchex && bash train.sh` flow, so the next agent doesn't think the camchex-cwd assumption is the only supported path.
- If we ever want true cwd-independence (so e.g. `--trainer.default_root_dir` can be passed project-root-relative), the right move is to resolve all paths inside `datamodule.py` / callbacks against an explicit `project_root` derived from `__file__`, and drop the chdir. Bigger change; not needed now.
