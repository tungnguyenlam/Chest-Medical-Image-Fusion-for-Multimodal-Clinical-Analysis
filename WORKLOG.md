## 2026-05-10 — make `camchex/main.py` runnable from project root

**Goal.** User reported `python camchex/main.py fit --config camchex/config.yaml --config camchex/config.local.yaml` from the project root crashes in `pd.read_csv` because `data.datamodule_cfg.train_df_path: ../data/data-camchex/03_mimic_train.csv` (and image paths like `images/p1x/...` in CSVs) are all written relative to `camchex/`, not the project root. They want a single command that works from project root without rewriting configs.

**Changes.**
- `camchex/main.py:1-22` — at module import time, walk `sys.argv` and rewrite any relative `--config <path>` / `--config=<path>` values to absolute paths against the original cwd, then `os.chdir()` into the script's own directory before constructing `LightningCLI`. Imports of `torch`/Lightning/`model.wrapper`/`dataset.datamodule` moved below the chdir block (cosmetic — they don't depend on cwd, but keeps the path-fixing logic at the top where it's visible).

**Reasoning.** Three candidates considered:

1. **Rewrite every path in `config.yaml` to be project-root-relative** and update `dataset/dataset.py` to resolve image paths against the project root. Touches the most files, breaks the existing `cd camchex && bash train.sh` workflow documented in `camchex/CLAUDE.md`, and `train.sh` itself launches with cwd=`camchex/` (`#SBATCH --output=../output/...`, `mkdir -p ../output/...`). Would force a coordinated change across SLURM scripts, rsync workflows, and per-machine `config.local.yaml` files on four machines.
2. **Wrapper script at project root** that just `cd`s into camchex and re-execs python. Works but adds a new entry point, doesn't help users who type the python command directly (which is what the user did).
3. **Chdir inside `main.py`** (chosen). One-file change, preserves the load-bearing assumption that cwd=`camchex/` at training time (config relative paths, CSV image paths, `mkdir -p ../output/...` in `train.sh`, callbacks importable as `callbacks.ema_callback.EMACallback` because `sys.path[0]` is the script dir). The `train.sh` flow is unaffected because it already cd's into camchex/ before invoking python, so the chdir is a no-op there.

The argv-rewriting step is necessary because `--config camchex/config.yaml` (the form the user typed) is project-root-relative; if we just chdir without rewriting, jsonargparse opens it relative to camchex/ and fails. Rewriting only `--config` (not arbitrary `--data.datamodule_cfg.train_df_path` overrides) is intentional — those latter values live inside the YAML and are already camchex-relative; rewriting them would break the YAML semantics.

**Assumptions.**
- `sys.path[0]` is set by Python to the directory containing `main.py` regardless of cwd, so the `from model.wrapper import ...` imports keep working after chdir. (True for `python camchex/main.py` invocation; would NOT be true for `python -m main` from project root, but that's not how the project is launched.)
- `config.local.yaml` doesn't exist on `kubuntu` (the dev machine where this edit was authored), so I couldn't smoke-test the exact failing command end-to-end. The error in the user's traceback was specifically `FileNotFoundError` on the train CSV path during `CaMCheXDataModule.__init__`, which is downstream of config parsing — confirming the config files themselves were found and the only remaining issue is the relative paths inside them. Chdir resolves that.
- All callsites that read paths from config do so via `pd.read_csv(...)` / `Image.open(...)` with no path normalization — verified by inspecting `dataset/datamodule.py:16` in the traceback. So plain chdir is sufficient; no path rewriting in the data layer needed.

**Gotchas.**
- The chdir runs at **import time**, before `if __name__ == "__main__"`. This is deliberate — `LightningCLI` parses argv inside its constructor, which is called from `cli_main()`, so the chdir must happen before that. But it means importing `camchex.main` as a module from elsewhere (e.g. tests) would have the side effect of changing cwd. Acceptable since nothing imports `main.py`; flag if that changes.
- Argv rewriting only handles `--config`. If a future user passes other path-bearing flags from project root (e.g. `--trainer.default_root_dir ../output/...`), those won't be rewritten and will resolve against camchex/ post-chdir. Not fixing speculatively.
- `train.sh` is unchanged. It still does `cd "$SLURM_SUBMIT_DIR"` and assumes the user submitted from `camchex/`. The new chdir in `main.py` is idempotent in that flow.
- `os.path.abspath(os.path.join(_ORIG_CWD, p))` — `os.path.join` discards `_ORIG_CWD` if `p` is already absolute, but we guard with `os.path.isabs(p)` first anyway for clarity.

**Follow-ups.**
- Smoke-test on the user's server (`hngoc@DESKTOP-UUC7V3K`) — they have `config.local.yaml` and the data files; this kubuntu machine doesn't.
- Consider documenting the new project-root invocation in `camchex/CLAUDE.md` ("Training" section) alongside the existing `cd camchex && bash train.sh` flow, so the next agent doesn't think the camchex-cwd assumption is the only supported path.
- If we ever want true cwd-independence (so e.g. `--trainer.default_root_dir` can be passed project-root-relative), the right move is to resolve all paths inside `datamodule.py` / callbacks against an explicit `project_root` derived from `__file__`, and drop the chdir. Bigger change; not needed now.

## 2026-05-10 — Fix path mismatch in step-03 filter (FileNotFoundError during validation)

**Goal.** Diagnose and fix `FileNotFoundError: Neither resized nor original image found: images/p15/p15411028/s50000800/...` raised by a `CaMCheXDataset` worker during validation sanity check on the user's WSL machine.

**Changes.**
- `camchex/data/03_filter_existing_images.py` — added a third element per source tuple, `base_dir_from_camchex_cwd` (e.g. `../data/MIMIC-CXR-JPG/files`), and rewrite the `path` column of every kept row to `<base_dir_from_camchex_cwd>/<rel>`. The filter still uses `base_dir` (project-root-relative) for the existence check because the script is invoked from project root.

**Reasoning.** The bug was a path-domain mismatch:
- Filter checked existence at `data/MIMIC-CXR-JPG/files/<rel>` (project-root-relative, since `03_*.py` runs from project root).
- Dataset opened `images/<rel>` from cwd `camchex/` (because `main.py` chdirs into `camchex/`). Resolves to `camchex/images/<rel>`, which only works if `camchex/images/` has per-folder symlinks into the source.
- Result: a row could pass the filter (file exists at the MIMIC base) yet still throw at load time (no matching symlink under `camchex/images/`). The user's machine has no `camchex/images/p15/...` so p15 rows that survived the filter blew up.

Two candidate fixes:
1. Add a `camchex/images → ../data/MIMIC-CXR-JPG/files` symlink on every machine. Reversible, but pushes a setup step onto each new machine and silently breaks if forgotten — same failure mode we just hit.
2. Have the filter rewrite `path` to point at the actual source dir using a camchex-relative prefix. Self-consistent: the same code that verified existence is the code that decides the saved path. Picked this — one source of truth, no per-machine setup.

**Assumptions.**
- Training cwd remains `camchex/` (enforced by the `os.chdir(_SCRIPT_DIR)` in `camchex/main.py`). If that ever changes, the `../` prefix in the saved paths becomes wrong.
- `camchex/` is rsynced as a unit and stays a sibling of `data/`, so `../data/...` resolves correctly on every machine.
- Per-folder Kaggle symlinks under `camchex/images/p1x/` are no longer load-bearing for `03_*.csv`-driven training (still fine for any code that opens `images/...` directly, e.g. ad-hoc scripts).
- The `_resized_1024.jpg` sibling-file fast path in `dataset.py` still works because we only change the directory prefix, not the filename.

**Gotchas.**
- The user must rerun `python camchex/data/03_filter_existing_images.py` to regenerate `03_*.csv` with the new path format. Existing 03 CSVs still contain `images/...` paths and will keep failing until regenerated.
- `02_*.csv` is unchanged, still uses `images/...`. Anything that reads `02_*.csv` directly (rare) still relies on the old convention.
- Existing `03_kaggle_*.csv` on disk on `kubuntu` is small (610 bytes ≈ header only) because the Kaggle base dir was empty at filter time — confirmed harmless, just means kaggle source isn't in use here.
- `filtered_df` now needs `.copy()` before assigning to `path` to avoid a `SettingWithCopyWarning`.

**Follow-ups.**
- User should rerun step 03 on every machine where 03 CSVs exist, then retry training.
- If anyone wants the old `images/...` convention back (e.g. to support training that mixes mimic + kaggle from the same CSV), we'd need a different shape — e.g. a per-row `source_tag` plus a base-dir lookup at dataset time. Not needed today.
- Could add a tiny sanity check to the start of `datamodule.py` that opens the first file in each split and fails fast with a clearer message, instead of the current "FileNotFoundError in worker" stack.
